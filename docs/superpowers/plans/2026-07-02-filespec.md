# FileSpec Implementation Plan (Phase 4)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** The file format description language: validate + render `statutory_file_formats.spec` documents to submission text files, seed the five government formats with byte-parity against the legacy formatter modules, add the file-format admin screen, switch the download path, then retire the legacy formatter modules and `SalaryNoteCalFunc`.

**Architecture:** `FullCircle.FileSpec` validates a spec map and renders it against row/context data. Field and filter expressions reuse the PayScript grammar/evaluator (new public expression-level API + string builtins). Aggregates (`sum`, `count`) are pre-substituted into the AST before evaluation, so the evaluator stays unchanged. `FullCircle.HR.Statutory` becomes a thin dispatcher: resolve the company's effective format version → render; legacy modules remain only until the parity gate passes.

**Tech Stack:** Phase 1–3 deliverables; Jason. No new deps.

## Global Constraints

- Phases 1–3 complete and green.
- **Byte parity is the gate**: for a fixture month, FileSpec output must equal the legacy formatter output line-for-line before any legacy code is deleted. Legacy modules are the oracle; never adjust an expected string to make FileSpec pass.
- Detail-row variables come from `HR.statutory_contributions/3` maps: stringified keys (`"name"`, `"id_no"`, `"tax_no"`, `"socso_no"`, `"epf_no"`, `"pay_month"`, `"pay_year"`, `"wages"`, plus one per statutory code). Decimals are converted to floats before evaluation.
- Header/footer context additionally gets `"employer_code"` (the per-agency code the CsvController already receives as `"code"`), `"pay_month"`, `"pay_year"`, `"company_name"`.
- Spec documents are maps with **string keys** (they come from jsonb / JSON textarea).
- Known pre-existing failures: 2 `pay_run_test` failures; credo not installed.

## FileSpec document reference (implemented by Task 2, validated by Task 2, edited in Task 4)

```json
{
  "renderer": "text",
  "line_ending": "\r\n",
  "delimiter": null,
  "sections": [
    {"kind": "header", "fields": [{"expr": "\"00\"", "width": 2}]},
    {"kind": "detail", "source": "statutory_rows",
     "filter": "socso_employee > 0 or socso_employer > 0",
     "sort": "name",
     "fields": [
       {"expr": "employer_code", "width": 12},
       {"expr": "replace(id_no, \"-\", \"\")", "width": 12},
       {"expr": "upper(name)", "width": 150},
       {"expr": "socso_employer + socso_employee", "width": 14, "format": "cents", "pad": "0", "align": "right"}
     ]},
    {"kind": "footer", "fields": [{"expr": "count()", "width": 7, "pad": "0", "align": "right"},
                                   {"expr": "sum(\"socso_employee\")", "width": 14, "format": "cents", "pad": "0", "align": "right"}]}
  ]
}
```

Rules: `sections` non-empty; `kind` in header/detail/footer; detail requires `source: "statutory_rows"`; `filter`/`sort` optional (sort = a row key, ascending); every field needs `expr`; fixed-width mode (field `width` present) uses `pad` (default `" "`) and `align` (default `"left"`); delimited mode (document `delimiter` set) forbids `width`; `format` in `"text"` (default) | `"cents"` (amount × 100, integer, no separators) | `"date:<Timex pattern>"`; `line_ending` default `"\r\n"`. Header/footer expressions may use `sum("col")`/`count()` over the **filtered** detail rows; detail expressions may not.

### Task 1: PayScript expression API + string builtins

**Files:**
- Modify: `lib/full_circle/pay_script.ex`, `lib/full_circle/pay_script/parser.ex`, `lib/full_circle/pay_script/evaluator.ex`, `lib/full_circle/pay_script/validator.ex`
- Test: extend `test/full_circle/pay_script_test.exs`, `test/full_circle/pay_script/evaluator_test.exs`, `test/full_circle/pay_script/validator_test.exs`

**Interfaces produced:**
- `PayScript.parse_expression(source) :: {:ok, expr_ast} | {:error, Error.t()}` — tokenizes, parses a **single expression**, errors on trailing tokens ("expected end of expression").
- `PayScript.eval_expression(expr_ast_or_source, context, env) :: {:ok, term} | {:error, Error.t()}` — raw value (number/string/boolean), **no** Decimal coercion (FileSpec formats strings too).
- `PayScript.validate_expression(source, schema) :: :ok | {:error, [Error.t()]}` — same checks as `validate/2` but for one expression; `schema.variables` required by caller (FileSpec passes row keys).
- New core builtins (evaluator + validator `@builtins`): `upper(s)`, `lower(s)`, `trim(s)` (strings, arity 1), `replace(s, from, to)` (arity 3, `String.replace/3`). Type errors follow the existing "expected a string, got ..." pattern.

Tests: `parse_expression("1 + 2")` ok; trailing garbage errors; `eval_expression(~s|upper(name)|, %{"name" => "ali"}, nil)` → `{:ok, "ALI"}`; `replace`/`trim`/`lower` happy + type-error cases; validator accepts the new builtins and rejects wrong arity.

- [ ] Steps: failing tests → verify → implement (parser: expose `parse_expression(tokens)` wrapper that calls the existing `parse_expr/1` then requires `[{:eof, _}]` remainder; evaluator: four new `eval_call` clauses using a `str1`/`str3` helper mirroring `num1`; validator: add to `@builtins`) → green (rerun the whole `test/full_circle/pay_script*` set) → commit `feat(payscript): expression-level api and string builtins`.

### Task 2: FileSpec validate + render

**Files:**
- Create: `lib/full_circle/file_spec.ex`
- Test: `test/full_circle/file_spec_test.exs`

**Interfaces produced:**
- `FileSpec.validate(spec :: map, variables :: [String.t()]) :: :ok | {:error, [String.t()]}` — structural rules from the reference above; every `expr`/`filter` through `PayScript.validate_expression` with `variables` (+ header/footer also allow the aggregate substitutes — validate aggregates by shape: `sum` takes one string literal, `count` takes zero args; check before substitution); errors prefixed `"section 2, field 3: ..."`.
- `FileSpec.render(spec, rows :: [map], header_ctx :: map) :: {:ok, String.t()} | {:error, String.t()}` — pipeline:
  1. Detail rows: stringify keys, Decimal→float, merge `header_ctx` into each row's context (so `employer_code` is usable in detail lines, as SOCSO needs).
  2. Apply `filter` (eval per row; non-boolean → error), then `sort`.
  3. For header/footer: precompute `count = length(filtered)` and each `sum("col")` over filtered rows; substitute into the AST: walk replacing `{:call, "count", []}` → `{:num, count}` and `{:call, "sum", [{:str, col}]}` → `{:num, sum}`.
  4. Render each field: `PayScript.eval_expression/3` with `env = nil` → apply `format` → fixed-width pad/align or collect for delimiter-join.
  5. Join fields per line, lines with `line_ending`, trailing line ending included (match legacy — verify against the oracle and encode the finding in the parity tests).
- Formatting: `cents` → `round(v * 100)` rendered as integer string (value must be a number); `date:<pattern>` → value must be a `Date`/string parseable date — detail rows carry no dates today except `service_since`; implement via `Timex.format!` and only accept `%Date{}` inputs; `text` → numbers rendered with `:erlang.float_to_binary(v, [:compact, decimals: 2])` when float, else `to_string`. Width overflow: **truncate right** for left-aligned text, error for numeric right-aligned fields (losing digits in an amount is corruption).

Tests (pure, async): validate catches every structural rule (one test each); render happy path fixed-width (assert exact padded string), delimited mode, filter excludes rows, sort orders, `sum`/`count` in footer, cents rounding (14.75 → `"1475"`), truncate/overflow behaviors, detail rows see `employer_code`.

- [ ] Steps: failing tests → verify → implement → green → commit `feat(filespec): spec validation and text renderer`.
